# コードレビュースキル

bookmarklyプロジェクトのコードを、フロントエンド・バックエンド・共有型それぞれの観点でレビューします。

## プロジェクト構成

```
apps/
  frontend/   Vue 3.5 + Vite 5 + TypeScript（<script setup>）
  backend/    Hono 4.4 + @hono/node-server + TypeScript
shared/
  types.ts    フロント・バック共通の型定義
```

---

## ワークフロー

### STEP 1: レイヤーの特定

レビュー対象のコードがどのレイヤーか判断する。

| パス / 特徴 | レイヤー |
|---|---|
| `apps/frontend/` / `.vue` ファイル | フロントエンド |
| `apps/backend/` / Hono ルート定義 | バックエンド |
| `shared/` / 型定義のみのファイル | 共有型 |
| 複数にまたがる | それぞれの観点で個別にレビュー |

### STEP 2: レイヤー別レビュー

該当レイヤーのチェックリストを適用する。

---

## レビュー基準

### 共通（TypeScript）

- [ ] `any` を使っていないか（代替: `unknown`、型ガード）
- [ ] `strictNullChecks` を前提とした null / undefined の扱いができているか
- [ ] 型推論に任せられる箇所に冗長な型注釈を書いていないか
- [ ] `type` と `interface` の使い分けが一貫しているか（オブジェクト形状は `type`、拡張が必要なら `interface`）
- [ ] `shared/types.ts` の既存型と重複していないか

---

### フロントエンド（Vue 3 + `<script setup>`）

#### Composition API
- [ ] `<script setup lang="ts">` を使っているか
- [ ] `ref` と `reactive` の使い分けが適切か（プリミティブ → `ref`、オブジェクト群 → `reactive`）
- [ ] `computed` を使うべき箇所でメソッド呼び出しになっていないか
- [ ] `watch` / `watchEffect` の使い分けが適切か
- [ ] `onUnmounted` でのクリーンアップが必要な副作用を放置していないか

#### 型定義
- [ ] `defineProps` / `defineEmits` に TypeScript の型を付けているか
- [ ] props はできるだけ narrowな型にしているか（`string` ではなく `'read' | 'unread'` など）

#### テンプレート
- [ ] テンプレート内にビジネスロジックが混入していないか
- [ ] `v-for` に `:key` を設定しているか（かつ index を避けているか）
- [ ] `v-if` と `v-for` を同一要素に併用していないか
- [ ] イベントハンドラが不必要に毎レンダー生成されていないか

#### 非同期処理
- [ ] `async/await` のエラーハンドリングが `try/catch` または `onErrorCaptured` で行われているか
- [ ] ローディング・エラー状態が管理されているか

#### 命名・スタイル
- [ ] コンポーネントは PascalCase か
- [ ] `defineExpose` の使用は最小限か（不必要に内部を公開していないか）

---

### バックエンド（Hono + Turso）

#### ルーティング
- [ ] ルートは RESTful な設計になっているか（動詞はHTTPメソッドで表現）
- [ ] パスは `/api/...` プレフィックスに統一されているか
- [ ] Hono の `c.json()` / `c.text()` / `c.status()` を適切に使っているか

#### レスポンス
- [ ] 成功・失敗それぞれに適切な HTTP ステータスコードを返しているか
  - 200 OK / 201 Created / 204 No Content
  - 400 Bad Request / 401 Unauthorized / 404 Not Found / 409 Conflict
  - 500 Internal Server Error
- [ ] レスポンス形状が `shared/types.ts` の型と一致しているか
- [ ] エラーレスポンスの形状が統一されているか

#### バリデーション
- [ ] リクエストボディ・クエリパラメータの入力バリデーションを行っているか
- [ ] バリデーション未通過時は 400 を返しているか

#### データベース（Turso / libsql）
- [ ] SQL はパラメータ化クエリ（`?` プレースホルダ）を使っているか（SQLインジェクション対策）
- [ ] `db.execute()` のエラーは `try/catch` でハンドリングされているか
- [ ] 不要なカラムを `SELECT *` で取得していないか

#### セキュリティ
- [ ] 環境変数（`TURSO_DATABASE_URL` など）をコードにハードコードしていないか
- [ ] 認証が必要なルートに認証チェックがあるか
- [ ] ユーザー入力を直接SQLやレスポンスに埋め込んでいないか

#### 型
- [ ] ルートハンドラの引数 `c` の型が適切に推論されているか
- [ ] DB の行データに対して型アサーションを使いすぎていないか（型ガードを優先）

---

### 共有型（`shared/types.ts`）

- [ ] フロント・バック両方で利用する型のみを置いているか
- [ ] 型名は PascalCase か
- [ ] `export` が適切についているか
- [ ] `any` / `object` を使っていないか

---

## 出力フォーマット

レビュー結果は以下の構造で出力する。

```
## レビュー結果：[ファイル名 or 機能名]

### 🔴 必須修正
（バグ・セキュリティリスク・型エラーなど、マージ前に直すべき問題）

### 🟡 改善推奨
（コード品質・可読性・設計上の改善点）

### 🟢 良い点
（適切な実装・工夫されている点を明示する）

### 💡 提案
（必須ではないが、検討に値するアイデア）
```

- 各指摘には **具体的なコード箇所**（行番号またはスニペット）と **修正案** を示す
- 指摘がないカテゴリは省略する
- 「良い点」は必ず1つ以上書く（指摘だけのレビューにしない）

---

## ヒント

- ユーザーがコードを貼り付けた場合は即座にレビューを開始する
- ファイルパスを指定された場合は Read ツールで内容を確認してからレビューする
- 「全体的にレビューして」と言われた場合は、変更されたファイルを `git diff` または直接読んで対象を特定する
- 複数ファイルにまたがる場合は、ファイルごとにセクションを分けて出力する
